Overall assessment
The architecture is fundamentally sound. Worker pool, async processing, inter-service HTTP client, price snapshot, and graceful shutdown are all correct senior-level patterns. The gaps are around service-to-service authentication, idempotency, total_amount calculation, and order ownership enforcement. All fixable before coding.
Quick gap summary:
🔴 Service-to-service auth not designed — how does Order Service authenticate with Inventory Service?
🔴 total_amount calculation missing — who calculates order total and when?
🟡 Order ownership not explicitly enforced — user can only see/cancel their own orders
🟡 Worker pool size as config — WORKER_POOL_SIZE in config but startup wiring not shown
✓ Correct 202 Accepted for async order processing ▶

Returning 202 Accepted immediately and processing asynchronously via worker pool is the correct pattern for long-running operations. The client gets a fast response and polls for status.

This directly mirrors how payment systems, shipping APIs, and large retailers handle order placement. ShopriteX processes thousands of orders — synchronous processing would make the API unacceptably slow.
✓ Correct Worker pool with configurable size and graceful drain ▶

Fixed pool of N goroutines + buffered job channel + WaitGroup for graceful shutdown is the production-grade Go concurrency pattern. Non-blocking Submit with ErrPoolFull is the correct backpressure mechanism.

Graceful drain on SIGTERM ensures no orders are lost mid-processing during deployments. This is what ShopriteX's JD specifically means by "goroutines and channels" and "concurrency-related defects".
✓ Correct Price snapshot in order_items ▶

Copying product_name and unit_price into order_items at order time is the correct retail pattern. Historical orders must never be affected by future price changes.

This is a classic senior interview question at retail companies. The correct answer is always: snapshot at order time. Never join to the products table for historical order data.
✓ Correct No FK from orders.user_id to users table ▶

Order Service stores user_id as a plain UUID without a foreign key to the users table (which lives in auth_db). Services own their own databases — cross-service FK constraints would couple the databases together.

Microservice principle: each service owns its data. Order Service validates user identity via JWT (Auth Service's responsibility), not via database FK. This allows Auth Service and Order Service to deploy independently.
✓ Correct Only PENDING orders can be cancelled ▶

The cancellation rule (only PENDING status) is the correct business logic. CONFIRMED orders require a refund flow. FAILED orders are already terminal. PROCESSING orders are mid-flight.

This is a deliberate business rule that prevents race conditions — if we allowed cancelling PROCESSING orders, we'd need to handle partial stock releases mid-reservation.
✓ Correct Context timeout on Inventory Service HTTP calls ▶

The inventory client has a configurable timeout (INVENTORY_CLIENT_TIMEOUT, default 5s). If Inventory Service is slow or down, the worker goroutine doesn't hang forever — it times out and marks the order as FAILED.

Without timeouts, a slow Inventory Service would gradually fill all worker goroutines with hung requests. The pool would become exhausted and new orders couldn't be processed. Timeouts prevent this cascading failure.
✓ Correct OrderFailed publishes release event back to Inventory ▶

When order processing fails after partial stock reservation, the processor calls inventory.Release() to return the reserved units. No stock is permanently locked by a failed order.

This prevents stock leakage — units reserved for failed orders must be returned to available. The release happens in the worker goroutine as part of the failure handling path.
✓ Correct select with default for non-blocking Submit ▶

Using select with a default case in Submit means job submission never blocks the HTTP handler. If the queue is full, the caller gets ErrPoolFull immediately — they can return 503 to the client rather than hanging.

This is the correct backpressure mechanism. Without default, a full channel would block the HTTP handler goroutine until a worker frees up — potentially for seconds. HTTP handlers must never block.
🔴 Missing Service-to-service authentication not designed ▶

The inventory client calls POST /inventory/reserve — but Inventory Service requires a JWT on all routes. How does Order Service authenticate itself to Inventory Service?

Not designed
Inventory client sends requests but has no Authorization header → 401 Unauthorized on every call
Solution
Order Service uses a dedicated service account JWT — issued at startup by Auth Service using a service account email/password stored in environment variables
In production, services authenticate with each other using service accounts or mTLS. For our project: Order Service logs in with a service account at startup, stores the JWT, and passes it on every call to Inventory Service. Refresh before expiry.
🔴 Missing total_amount calculation not specified ▶

The orders table has total_amount but we never specified who calculates it, when, and from what data.

Problem
Client sends product IDs and quantities. We don't have prices in the Order Service. Where do prices come from?
Solution
Client sends product_id, quantity, AND unit_price in the order request. Order Service validates that unit_price > 0 but trusts the client price. Total = sum of (quantity × unit_price) for all items.
// CreateOrderRequest — client sends prices
type OrderItemRequest struct {
    ProductID   string  `json:"product_id"   validate:"required"`
    ProductName string  `json:"product_name" validate:"required"`
    Quantity    int     `json:"quantity"     validate:"required,min=1"`
    UnitPrice   float64 `json:"unit_price"   validate:"required,min=0.01"`
}

// Service calculates total
total := 0.0
for _, item := range req.Items {
    total += float64(item.Quantity) * item.UnitPrice
}
In a real system, Order Service would call Inventory Service to get the authoritative price. For our project, the client provides prices — simpler and sufficient for demonstrating the architecture.
🟡 Refine Order ownership enforcement needs explicit design ▶

GET /orders/:id and PATCH /orders/:id/cancel must verify the order belongs to the requesting user. This was implicit in the original design but needs to be explicit in the repository and service layers.

// Repository — FindByID must include user_id check
func (r *postgresOrderRepo) FindByID(ctx context.Context, orderID, userID string) (*domain.Order, error) {
    query := `
        SELECT id, user_id, status, total_amount, created_at, updated_at
        FROM orders
        WHERE id = $1 AND user_id = $2   ← ownership enforced in SQL
    `
    // Returns ErrOrderNotFound if order exists but belongs to different user
    // This prevents user A from seeing user B's orders
}
Enforcing ownership in SQL is safer than doing it in the service layer. A service-layer check requires two queries (find then check). The SQL WHERE clause does it in one query and prevents information leakage — user A doesn't even know user B's order exists.
🟡 Refine Worker pool startup wiring needs clarification ▶

The worker pool must be started BEFORE the HTTP server begins accepting requests, and shut down AFTER the HTTP server drains. The startup sequence in main.go needs to explicitly show this ordering.

// Correct startup order in main.go:
pool := worker.NewWorkerPool(cfg.WorkerPoolSize, orderService)
pool.Start(ctx)           // ← goroutines running, ready to accept jobs

srv := server.New(...)
srv.Run()                 // ← HTTP server starts, handlers can Submit to pool

// Correct shutdown order in server.go:
httpServer.Shutdown(ctx)  // ← stop accepting requests, drain handlers
pool.Shutdown()           // ← drain job channel, wait for workers
db.Close()                // ← close DB after all workers done

+ Add internal/client/auth_client.go — token manager ▶

A lightweight client that Order Service uses to get and refresh its service account JWT. Called once at startup, then periodically to refresh before the 15-minute expiry.

type ServiceTokenManager struct {
    authURL  string
    email    string
    password string
    token    string
    expiry   time.Time
    mu       sync.RWMutex  // protects concurrent token reads
}

func (m *ServiceTokenManager) GetToken(ctx context.Context) (string, error) {
    m.mu.RLock()
    if time.Now().Before(m.expiry.Add(-1 * time.Minute)) {
        defer m.mu.RUnlock()
        return m.token, nil  // token still valid
    }
    m.mu.RUnlock()
    return m.refreshToken(ctx)  // refresh token
}
