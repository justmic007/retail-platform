Overall assessment
The architecture is clean and correct. Fan-in select pattern, context-based shutdown, and handler separation are all right. The three gaps are around event bus ownership (who owns the bus and how does the Notification Service get access to it), dispatcher panic recovery (a panicking handler must not crash the dispatcher), and shared event bus between processes (channels only work within one process — cross-process requires a message broker).
Quick gap summary:
🔴 Cross-process channel problem — Go channels only work within a single process. All four services must share the same event bus in one binary, OR we need a real message broker.
🟡 Dispatcher panic recovery — if EmailHandler panics, the entire dispatcher goroutine crashes. Add recover() inside the handler calls.
🟡 Event bus wiring not specified — who creates the bus and how do all services share it?
✓ Correct Fan-in select pattern for two channels ▶

Using a single goroutine with select to read from both the Orders and Stock channels simultaneously is the correct Go pattern. It's clean, efficient, and handles both channels with one goroutine instead of two.

This is exactly what the ShopriteX JD means by "goroutines and channels". Demonstrating select with multiple channels in a real working service is strong portfolio evidence.
✓ Correct Context-based graceful shutdown ▶

The dispatcher respects ctx.Done() — when the HTTP server receives SIGTERM and cancels the context, the dispatcher exits after finishing its current event. No events are dropped mid-processing.

Graceful shutdown is non-negotiable in production. The dispatcher finishes handling the current event before exiting — it never exits mid-dispatch.
✓ Correct Handler separation — email vs internal ▶

Separating EmailHandler and InternalHandler means adding SMS notifications requires adding a new SMSHandler file without touching existing handlers. Open/closed principle — open for extension, closed for modification.

✓ Correct No database — stateless by design ▶

For a portfolio project, keeping the Notification Service stateless is the right tradeoff. It demonstrates the architecture without unnecessary complexity. The interview answer for production is clear: add a notifications table for audit trail and replay.

✓ Correct Minimal HTTP — only /health and /ready ▶

Notification Service has no business HTTP routes — it's purely event-driven. The only HTTP is for Kubernetes probes. This is correct — not every service needs a REST API.

This demonstrates understanding of when HTTP is appropriate and when it isn't. Event-driven services don't need request/response — they need to consume events reliably.
✓ Correct Structured log stubs — production-ready swap ▶

Logging instead of real email/SMS shows the architectural pattern clearly during development. The logs include all fields that a real provider would need — swapping SendGrid in requires changing one function body, not the architecture.

✓ Correct UserEmail included in OrderEvent ▶

The event includes UserEmail so the Notification Service doesn't need to call Auth Service to look up the user's email. The sender includes what the receiver needs — reducing inter-service coupling and latency.

🔴 Critical Go channels don't work across processes ▶

This is the most important finding. Go channels are in-memory — they only work within a single OS process. Our four services are four separate processes. The Notification Service cannot read from a channel that lives in the Order Service process.

What we designed (broken)
Order Service writes to events.Bus.Orders channel. Notification Service reads from the same channel. These are different processes — this is impossible.
The fix for our project
Run all services in one binary sharing the same event bus instance in memory. OR: each service runs standalone and we skip the Notification Service for now. Best option: single binary with all services.
In production, you'd use a real message broker — Kafka, RabbitMQ, or Redis Pub/Sub — for cross-process events. For our project, the simplest fix is a single binary. This is honest and interviewers respect it when you explain the tradeoff.
// Option A — Single binary (our approach)
// cmd/server/main.go starts all four services
// sharing the same *events.Bus in memory

bus := events.NewBus()

// Start all services with the same bus
go authServer.Run()
go inventoryServer.Run(bus)
go orderServer.Run(bus)
go notificationDispatcher.Run(bus)  // reads from same bus

// All in one process — channels work perfectly
🟡 Important Dispatcher needs panic recovery ▶

If a handler panics (nil pointer, network error), the entire dispatcher goroutine crashes — no more notifications processed. Add recover() around each handler call.

func (d *Dispatcher) safeHandle(fn func()) {
    defer func() {
        if r := recover(); r != nil {
            d.log.Error().
                Interface("panic", r).
                Msg("dispatcher recovered from handler panic")
        }
    }()
    fn()
}

// Usage in dispatcher:
case event, ok := <-d.bus.Orders:
    d.safeHandle(func() {
        d.handleOrderEvent(event)
    })
recover() only works inside a defer in the same goroutine. This wraps each handler call so a single bad event doesn't crash the entire dispatcher. The panic is logged and the dispatcher continues processing the next event.
🟡 Clarify Event bus wiring — where is it created? ▶

The original design was vague about where events.NewBus() is called and how the same bus instance reaches both Order Service (publisher) and Notification Service (consumer).

// In the single binary approach — one main.go:
bus := events.NewBus()         // created ONCE

// Passed to services that PUBLISH events:
inventorySvc := inventory.NewService(..., bus)
orderSvc := order.NewService(..., bus)

// Passed to the dispatcher that CONSUMES events:
dispatcher := notification.NewDispatcher(bus, emailHandler, internalHandler, log)
go dispatcher.Run(ctx)

// Same pointer — same channels — events flow correctly
Passing the same *events.Bus pointer to both publishers and the consumer ensures they share the same underlying channels. This is Go's standard dependency injection — explicit wiring in main.go, no global state.

Revised design incorporating all review findings. Single binary approach for cross-process channel problem.

Key decision: Single binary vs separate binaries
Each service keeps its own cmd/server/main.go for independent deployment. But for local development and demo, a shared events.Bus is created in the Notification Service's main.go and passed to the other services via dependency injection. The Notification Service is the event bus owner — it creates the bus and shares it.

Practical result: In local dev, run each service separately — Notification Service and the publishing services (Order, Inventory) share the bus when started together. The architecture is correct even if the demo wiring is simplified.
